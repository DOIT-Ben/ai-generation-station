## 执行记录
### TODO 1 复核现状
- 已复核 round-12 后的聊天页基线，确认下一阶段最缺的不是消息层细节，而是会话层面的“管理语义”: 当前列表只有时间排序，缺少重点会话、稍后处理和持续整理入口。
- 已确认本轮仍不动后端，在现有会话列表、筛选和本地偏好机制上直接扩展工作流能力。

#### 单任务复盘
- 新问题：当会话数量开始累积后，纯时间倒序会让“真正重要的对话”和“临时用完就走的对话”挤在一起，长期使用会越来越费眼。
- 边界条件：这轮不能依赖新的后端字段，否则会把当前可落地的前端工作流打断。
- 遗漏补齐：需要让同一套本地状态兼容搜索、筛选、分组、归档和删除，不然很容易出现列表不同步。

### TODO 2 增加本地会话工作流状态
- 增加了按用户维度持久化的本地会话工作流状态，支持 `重点会话` 与 `稍后处理` 两类标记。
- 本地状态通过 `localStorage` 与当前登录用户绑定，登录后自动水合，退出登录时会重置内存态，避免跨账号串状态。
- 永久删除已归档会话时，会同步移除相关本地工作流标记，避免留下悬空状态。

#### 单任务复盘
- 新问题：本地状态如果不按用户维度隔离，很容易在多人共用浏览器时产生误导。
- 边界条件：重点和稍后必须是“前端增强层”，不能污染现有会话对象和远端排序逻辑。
- 遗漏补齐：删除已归档会话后如果不清理本地标记，后续恢复或筛选会出现幽灵状态，所以补了统一清理入口。

### TODO 3 改造会话列表与筛选
- 筛选栏增加了 `重点` 与 `稍后` 两个新维度，搜索反馈与摘要文案也同步覆盖这些模式。
- 会话列表排序改为“重点优先、普通居中、稍后下沉”，并增加“重点会话 / 稍后处理”分区，让列表从纯时间轴升级为有工作流层次的结构。
- 会话行标签与主对话上下文条同步展示 `重点会话 / 稍后处理` 状态，进入对话后也能保持状态连续感。

#### 单任务复盘
- 新问题：会话工作流的价值不在单个标记按钮，而在标记后列表秩序是否真的变化。
- 边界条件：不能让重点与稍后把原本的“今天 / 最近”时间感完全打散，所以采用“重点前置、稍后后置、普通仍按时间分组”的混合结构。
- 遗漏补齐：已归档列表也需要兼容重点/稍后筛选，否则用户会看到摘要计数和筛选结果对不上。

### TODO 4 优化会话级快捷操作
- 会话行内动作增加了 `设为重点 / 取消重点` 与 `稍后处理 / 取消稍后`，并保留原有改名和归档动作，形成完整的列表内整理闭环。
- 摘要工具区新增 `只看重点` 和 `只看稍后` 快捷入口，减少用户每次都去筛选栏里点来点去。
- 同步压紧了会话行内动作与标签样式，让新增动作不会把桌面和移动端布局再次撑乱。

#### 单任务复盘
- 新问题：如果工作流状态只能在筛选里看见，用户依旧要先“想到功能”再去找入口；高频入口必须直接落在列表行里。
- 边界条件：行内动作再多也不能盖住会话内容，所以改成更紧凑的胶囊按钮并允许桌面端两行换行。
- 遗漏补齐：除了列表行，摘要工具区也要有工作流入口，否则用户无法从全局视角快速聚焦重点或稍后任务。
