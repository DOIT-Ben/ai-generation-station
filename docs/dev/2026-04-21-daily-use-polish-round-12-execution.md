## 执行记录
### TODO 1 复核现状
- 已复核 round-11 后的聊天页基线，确认主要剩余问题不在功能缺失，而在高频使用时的细小摩擦：消息尾部动作过平、移动端软键盘场景缺少视口补偿、空白态到继续追问的阶段反馈仍然偏弱。
- 已确认本轮仍只处理聊天页前端与静态测试，不触碰后端协议，不带入外部脏文件 `public/js/app-shell.js`。

#### 单任务复盘
- 新问题：round-11 已把结构收口，但“长期天天用”的问题更多体现在每次发送、每次滚动、每次改写时的细节，而不是大块布局。
- 边界条件：这轮不能靠新增很多区块来制造“体验升级”，否则会再次把输入区做重。
- 遗漏补齐：把阶段反馈优先落在现有输入区文案和状态上，而不是再堆一块新的提示面板。

### TODO 2 重构消息级操作区
- 将 AI 回复尾部动作改为分层结构：保留复制与重写/重试为主动作，把多版本导航收纳进可展开的版本面板。
- 版本面板支持“查看版本 x/y -> 收起版本”的单击切换，并在展开时显示当前版本说明与上下切换控件，避免所有版本按钮长期平铺在每条回复下面。
- 同时补了只保留一个展开面板的逻辑，减少长会话中多个版本区同时展开带来的视觉拖尾。

#### 单任务复盘
- 新问题：多版本回复真正的问题不是缺按钮，而是按钮始终裸露在正文下方，导致长会话里每条回复尾部都像一个独立工具条。
- 边界条件：动作收纳不能牺牲“复制”和“快速重写”的即时性，因此主动作继续常驻。
- 遗漏补齐：版本展开切换不应该通过整段消息重渲染来实现，否则会打断阅读，所以改成保存状态并直接同步当前消息 DOM。

### TODO 3 增强移动端细节
- 增加了聊天视口同步逻辑，监听 `visualViewport` 与输入框 focus/blur，在手机软键盘弹起时计算键盘占位并同步到 CSS 变量。
- “回到最新回复”按钮现在会自动抬升到键盘上方，避免在输入过程中被遮住；输入区在移动端 composing 状态下也会加上轻微浮层阴影，增强边界感。
- 同时补了输入框 focus 时的可视区对齐逻辑，让用户在手机端继续输入时更容易留在聊天底部。

#### 单任务复盘
- 新问题：移动端聊天体验的核心不是“能不能输入”，而是键盘弹起后底部控件是否还在一个稳定、可预期的位置。
- 边界条件：键盘视口补偿必须只在窄屏且输入聚焦时生效，不能影响桌面端和普通滚动。
- 遗漏补齐：滚动按钮在用户上滑阅读时不只是显隐问题，还要在流式生成中改成“有新回复，回到最新”，让状态更明确。

### TODO 4 优化空白态与继续追问节奏
- 在不增加新面板的前提下，重写了输入区的主文案与辅助文案，让它根据“空白会话 / 第一条草稿 / 第一轮刚开始 / 已有上下文继续追问 / 正在生成 / 队列中”自动切换。
- suggestion strip 的追问标题和说明也补了阶段差异，第一轮结束后会更强调“补限制条件、目标结果、风格偏好”，而不是一律使用同一套追问文案。
- starter footer 也更新为更明确的节奏说明，告诉用户发出第一句后会自动切到继续追问模式。

#### 单任务复盘
- 新问题：真正的“节奏感”不是多一个引导模块，而是同一个输入区在不同阶段说的话是否像一个连续的助手。
- 边界条件：空白态和继续追问态必须共用现有结构，否则用户一发送第一条就会感觉界面在“跳皮肤”。
- 遗漏补齐：正在生成和队列状态也属于阶段的一部分，不能只考虑空白态与已聊天态两端。

### TODO 5 回归与收尾
- 已完成语法检查、页面结构断言、前端烟测、视觉基线更新与视觉回归。
- 已确认本轮提交范围只包含 `public/js/app.js`、`public/css/style.css`、`test-page-markup.js` 与 round-12 文档，继续排除 `public/js/app-shell.js`。

#### 单任务复盘
- 新问题：这类体验打磨轮次最容易漏掉的是“代码没报错，但视觉与状态联动退回去了”，所以必须保留功能回归和视觉回归双轨。
- 边界条件：视觉基线更新必须与最终样式同步，不能在更新基线后继续做额外视觉改动。
- 遗漏补齐：移动端视口与消息动作收纳都已补进静态断言，避免后续被误删。
