# 2026-04-20 Chat UX Polish Plan

## 目标
- 在已完成真实流式输出与回复版本切换的基础上，补齐“长期天天用”所需的剩余聊天体验细节。
- 将聊天界面收敛成更稳定的工作台风格：信息密度更高，但干扰更低，状态反馈更明确。
- 保持现有后端流式契约不变，主要通过前端状态机、文案和样式完成升级。

## 设计取向
- 视觉方向：专业工作台，而不是营销落地页。重点是秩序、反馈、阅读舒适度。
- 交互方向：用户始终知道系统当前在做什么，失败时知道下一步怎么恢复。
- 信息架构方向：会话列表承担“找回上下文”，消息卡承担“完成当前任务”。

## 顺序化 TODO
1. 审查当前聊天前端状态、消息动作区、会话列表结构、底部继续工作条和移动端样式边界。
2. 为聊天发送链路补充中止能力、失败态保留与重试入口，解决“卡住只能等”和“失败后无恢复动作”问题。
3. 增强消息级状态反馈，包括生成中、失败、复制成功、已切换版本等可感知提示。
4. 优化会话列表信息架构，增加摘要预览、时间分组和更明确的当前会话高亮。
5. 继续弱化底部“继续上次工作”条，让它更像辅助工具条而不是主内容卡片。
6. 细修 AI 回复阅读区与移动端动作布局，确保长回复、版本切换、失败恢复在小屏上仍可用。
7. 更新静态断言与开发过程文档，执行语法检查与前端烟测，修复回归问题后提交。

## 验收标准
- 生成中的回复可被用户主动停止，且停止后界面不会丢失当前上下文。
- 生成失败后，用户能直接在消息处理解原因并进行重试，而不是重新手工组织上下文。
- 会话列表可快速区分“今天 / 昨天 / 更早”的记录，并看到每个会话的大致内容。
- AI 回复动作区更清楚，用户能明显感知复制、重写、版本切换、失败重试的结果。
- 底部“继续上次工作”不再抢占视觉主导权，但仍可用于恢复草稿。
- 移动端下消息动作、底部工具条、会话列表和输入区不拥挤、不遮挡、不跳动。
