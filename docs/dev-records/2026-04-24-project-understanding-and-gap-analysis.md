## 验证结果
- 项目当前认知：
  - 这是一个“AI 内容生成工作台”项目，覆盖 AI 对话、歌词、音乐、图片、语音、翻唱，并扩展了认证中心、个人中心、管理后台、历史记录、模板库和会话持久化。
  - 后端采用 Node 原生 `http` + 自写路由分发；数据状态集中在 SQLite；前端采用原生 HTML + CSS + 大体量浏览器脚本。
  - 当前项目不是“功能很少的 demo”，已经开始承担真实产品工作台职责，因此安全、维护性和验证链质量的要求应按产品化标准看待。

- 当前主要不足，按影响排序如下：

### 1. 当前必修：`/output/*` 存在任意文件读取风险
- 位置：`server\routes\local.js:121-123`
- 现象：
  - 该路由直接将 `req.url.replace('/output/', '')` 结果拼到 `path.join(OUTPUT_DIR, filename)`。
  - 没有像静态资源路由那样做归一化后根目录约束检查。
  - 该路由也没有要求登录态。
- 证据：
  - 本轮临时验证脚本请求 `/output/../package.json`，实际返回 `200` 且读到项目 `package.json` 内容。
- 风险：
  - 外部请求可以越过 `output` 目录读取仓库内其他文件。
  - 若配合环境或配置文件命名猜测，可能进一步扩大为敏感信息泄露。
- 归类：当前必修。

### 2. 当前必修：TTS 路由错误路径会抛出 `reject is not defined`
- 位置：`server\routes\service.js:200-292`
- 现象：
  - `new Promise((resolve) => { ... })` 只声明了 `resolve`，但内部多处调用 `reject(...)` 和 `apiReq.on('error', reject)`。
- 证据：
  - 本轮通过注入假 `https` 客户端模拟上游失败，调用 `/api/tts` 后返回 `500`，错误正文为 `reject is not defined`。
- 风险：
  - 只要 TTS 上游返回异常、无音频或网络错误，服务端就不会返回真实业务错误，而是抛出实现错误。
  - 该问题说明对应异常路径缺少测试覆盖。
- 归类：当前必修。

### 3. 当前必修：默认账号口令被硬编码，且前端公开暴露了同一组凭据常量
- 位置：
  - `server\config.js:87`
  - `public\js\app-shell.js:10-12`
  - `public\js\app-shell.js:581`
  - `public\js\app-shell.js:1357-1358`
- 现象：
  - 服务端在未显式配置时默认使用 `APP_PASSWORD = 'AIGS2026!'`。
  - 前端 `app-shell.js` 内保留 `AUTH` 常量、`authenticate` 函数，并将 `AUTH` 导出给浏览器环境。
  - `app-shell.js` 被 `public\auth\index.html`、`public\admin\index.html`、`public\account\index.html` 统一加载。
- 风险：
  - 即便当前真实鉴权可能主要依赖服务端 session，这组前端暴露的默认凭据也会制造错误安全边界。
  - 一旦环境配置缺失或初始化流程出错，项目存在默认弱口令落地风险。
  - 即使不直接被使用，公开暴露这类常量也会放大误用和维护误判。
- 归类：当前必修。

### 4. 重要但可排在安全问题之后：核心实现过度集中，维护成本过高
- 位置与规模：
  - `public\js\app.js`：约 `250714` 字节，`6546` 行
  - `public\js\app-shell.js`：约 `71578` 字节，`1373` 行
  - `server\state-store.js`：约 `91462` 字节，`2278` 行
  - `server\routes\state.js`：约 `51141` 字节，`1360` 行
- 现象：
  - 前端工作台逻辑、聊天状态、模板交互、会话管理、工作区状态等全部堆在 `app.js`。
  - 服务端认证、用户管理、邀请、找回密码、审计等全部堆在 `state.js`。
  - 数据表结构、SQL、业务拼装、历史记录、会话流转、模板、审计又集中在 `state-store.js`。
- 风险：
  - 单点改动的回归面过大。
  - 新问题定位越来越依赖个人记忆，不利于协作和后续拆分。
  - 这种体量下，即使测试存在，也容易出现“修一个点，碰三个边界”的情况。
- 归类：规划缺失，需补分层与拆分规划。

### 5. 重要问题：前端公共壳体过重，所有页面都在背工作台级负担
- 位置：
  - `public\auth\index.html:143`
  - `public\admin\index.html:213`
  - `public\account\index.html:126`
  - `public\js\app-shell.js`
- 现象：
  - 认证页、后台页、个人中心都统一加载 `app-shell.js`。
  - `app-shell.js` 内包含大量模板库、会话筛选、远程持久化封装和历史相关能力。
  - 根项目没有构建、分包、压缩步骤，`package.json` 里也没有 `build` 或 `lint` 脚本。
- 风险：
  - 非工作台页面也要下载和解析大量与当前页面无关的逻辑与静态内容。
  - 项目越长，首屏负担和修改耦合越重。
  - 共享壳体会让“公共基础设施”和“业务内容库”互相污染。
- 归类：可延后，但建议尽早纳入拆分计划。

### 6. 中度问题：验证链数量不少，但组织方式偏脆弱
- 位置：
  - `package.json:4-27`
  - 根目录多个 `test-*.js`
- 现象：
  - 项目已有较多验证脚本，但主要通过脚本串联执行。
  - 缺少统一测试框架、缺少 lint、缺少构建期校验。
  - 本轮发现的 TTS 错误路径问题说明“有测试”不等于“关键异常路径已覆盖”。
- 风险：
  - 新增测试和定位失败成本越来越高。
  - 结果可读性、分层覆盖度、失败定位体验都不如标准测试框架。
- 归类：可延后，但应纳入工程治理 TODO。

### 7. 中度问题：项目说明和历史文档存在漂移
- 位置：
  - 仓库根目录缺少 `README.md`
  - `TEST-REPORT.md:94,157,160,189`
- 现象：
  - 根目录没有统一项目说明。
  - `TEST-REPORT.md` 仍引用不存在的 `frontend\js\app.js`、`js\models\api.js` 等旧结构。
  - 文档里仍给出“良好，可上线”的结论，但当前代码已引入新的认证、后台、状态库和安全边界，结论明显滞后。
- 风险：
  - 新人或未来自己会先被旧结构误导。
  - 文档结论与现状不一致时，会削弱后续排查和评审可信度。
- 归类：规划缺失，可延后处理，但不应长期放任。

## 复盘
- 这次项目了解后，最大的结论不是“功能还不够多”，而是“产品化速度已经快过工程整理速度”。
- 当前仓库最突出的不是单点功能缺失，而是三类失衡：
  - 安全边界已经进入真实产品阶段，但仍保留了 demo 式开放路由和默认口令痕迹。
  - 功能已经扩展到认证、后台、模板、会话、审计，但代码组织仍主要依赖超大文件承载。
  - 测试脚本数量不少，但还没形成能稳定守住异常路径和结构演进的验证体系。
- 新问题分类：
  - 当前必修：
    - `/output/*` 目录穿越读取
    - `/api/tts` 错误路径 `reject is not defined`
    - 默认口令与前端暴露凭据常量
  - 可延后：
    - 超大文件拆分
    - 前端公共壳体减重
    - 测试体系标准化
    - README 与测试文档回写
- 若后续要继续推进，合理顺序应是：
  1. 先修安全边界和确定性缺陷。
  2. 再补验证，确保这些问题被测试固定住。
  3. 最后按主线拆分前后端超大文件，而不是一边修漏洞一边做大范围重构。
