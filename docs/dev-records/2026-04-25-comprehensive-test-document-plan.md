# 2026-04-25 Comprehensive Test Document Plan

## 目标
- 在进入开发和修复前，先编写一份全面、完整、可直接落地的测试文档。
- 明确本项目需要测试的范围、测试类型、测试用例、执行顺序、验收标准和命令。
- 将测试文档作为后续修复工作的前置依据。

## 范围
- 覆盖当前项目的核心模块：
  - 认证、会话、CSRF、权限边界。
  - 聊天模型列表、聊天接口、流式回复、会话持久化。
  - 多模态生成：歌词、音乐、图片、语音、翻唱、上传、任务状态。
  - 文件输出、资源访问、历史记录、偏好设置、模板库、管理员后台。
  - UI 主题、护眼模式、下拉框、响应式布局、样式契约。
  - 配置、安全、容量、回归与发布前检查。
- 当前只写文档，不修改业务代码和测试代码。

## 假设
- 后续修复将优先处理代码审查发现的 P0/P1 问题。
- 测试文档需要兼顾已有测试脚本和新增测试缺口。
- 测试命令以当前 Node 脚本体系为主。

## 风险
- 项目功能面较宽，测试文档如果过粗会无法落地；因此用模块化清单和明确命令拆分。
- 一些外部 API 测试可能消耗额度或受网络影响，需要区分 mock、本地集成、真实联调。
- 当前工作区已有测试失败项，文档必须记录为已知状态而不是忽略。
