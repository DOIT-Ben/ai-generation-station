# 音视频内容提取界面规划记录

## 目标
- 在当前项目中新增一个上传音视频并自动提取内容的界面，但仅在需求定义清晰后进入实现。

## 范围
- 优先复用现有单页 Tab 架构、现有上传交互、现有前端样式体系和现有 Node 服务。
- 当前阶段只做需求澄清、现状梳理和实施规划，不直接编码。

## 已确认现状
- 当前项目前端主界面位于 `public\index.html`，通过 Tab 方式承载多个功能页。
- 当前项目前端主逻辑位于 `public\js\app.js`。
- 当前后端由 `server\index.js` 统一提供静态页面和 API。
- 现有后端已有 `\api\upload` 上传接口，可保存 base64 文件到 `output` 目录。
- 当前已有音频上传交互可复用，参考 `歌声翻唱` 页。

## 关键待澄清点
- “自动提取音视频里面的内容” 的目标结果尚不明确，至少存在以下几种完全不同的定义：
  - 语音转文字：提取音频中的 spoken content，输出逐字稿。
  - 视频画面内容提取：抽帧后做 OCR 或画面描述。
  - 音视频综合摘要：同时提取语音文字、画面文字、摘要结论。
- 输出形式也尚未明确：
  - 只显示纯文本结果。
  - 显示时间轴片段。
  - 支持复制、导出、继续用于聊天或歌词等功能。

## 假设
- 将采用最小方案，不新增独立前端工程，不拆新服务。
- 新能力优先落在现有主界面新增一个 Tab，而不是另起页面。
- 若用户仅需要“内容提取”，默认不扩展到复杂工作流编排。

## 风险
- 若在“提取内容”定义不清的情况下直接实现，极易出现输出形态与用户目标不一致。
- 视频内容提取相比音频转文字复杂度高很多，可能涉及转码、抽帧、OCR、摘要多段流程。
- 上传文件体积和格式限制若不先定义，后端和前端的约束会反复返工。

## 当前范围收敛
- 本轮仅开发“语音转文字”功能壳子。
- 不接第三方 API。
- 不实现真实转写。
- 只完成前端界面、上传交互占位、按钮占位、结果展示占位。

## 补充方案：第三方转写 API 调用方式
- 目标已收敛为“语音转文字”。
- MiniMax 当前不作为转写提供方。
- 若走第三方 API，推荐统一由后端调用，不让前端直接暴露第三方密钥。

### API 接入通用链路
1. 前端上传音频或视频到本项目后端。
2. 后端保存文件。
3. 若是视频，后端用 `ffmpeg` 抽取音轨，转成单独音频文件。
4. 后端将音频文件发送给第三方 ASR API。
5. 后端拿到转写结果后，整理成统一 JSON 返回给前端。

### 常见可选 API
- OpenAI Speech-to-Text：
  - 文档：`https://platform.openai.com/docs/guides/speech-to-text`
  - 接口：`POST https://api.openai.com/v1/audio/transcriptions`
  - 适合快速接入，支持常见音视频格式。
- AssemblyAI Speech-to-Text：
  - 文档：`https://www.assemblyai.com/docs/speech-to-text`
  - 适合做更强的分段、时间戳、说话人等扩展。

### 当前仓库最小建议
- 若要最快上线 API 版，优先用 OpenAI Speech-to-Text。
- 若要更强的转写增强能力，再评估 AssemblyAI。
