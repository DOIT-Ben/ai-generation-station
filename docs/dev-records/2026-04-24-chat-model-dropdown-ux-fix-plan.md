# 聊天模型下拉框 UX 修复记录

## 目标
- 修复聊天区模型下拉框“可见高度不足、滚动不可发现、推荐模型与实际可用模型脱节”的问题。

## 范围
- 仅处理聊天模型下拉框相关实现。
- 允许修改的文件限定为：
  - `public\css\style.css`
  - `public\js\app.js`
  - `server\routes\service.js`
  - 与本任务直接相关的现有测试文件

## 假设
- 当前用户反馈对应的是聊天区 `chat-model-dropdown`，不是配置面板里的小型下拉框。
- 后端 `/api\service\models` 返回的数据已经代表“当前环境可展示的模型候选”，前端推荐区应基于该实际返回值生成，而不是展示不存在的推荐项。
- 本轮按最小方案处理，不新增复杂折叠交互；搜索框仅在改动成本可控且不扩大范围时再评估。

## 风险
- 统一修改 `.custom-dropdown .dropdown-menu` 可能影响其他大号下拉框样式，需要核对是否仅聊天模型下拉框使用该样式。
- 前端重建下拉内容后，若选择态同步逻辑依赖旧节点引用，可能出现 active 状态不同步。
- 推荐模型过滤逻辑若放在错误层级，可能导致默认模型不在推荐区但仍能被选择，需要验证默认值回显。

## 完成标准
- 聊天模型下拉框打开后，可视高度明显增加，用户无需立刻滚动即可看到更多模型。
- 下拉框在存在更多内容时具备清晰的滚动暗示或视觉提示。
- 当前选中模型在列表中有明显识别，不仅仅依赖文字加粗。
- 分组标题识别度高于现状，不易被忽略。
- 推荐模型区不会显示当前接口未返回的模型。
- 现有功能不回退：模型选择、默认值回显、推荐分组渲染仍正常。
